The problem

Every Kafka cluster has a topic graveyard. Nobody dares delete.

Topics outlive the services that produced them. Storage cost grows linearly with topic count, but the value per topic decays. Without shared evidence, nobody pulls the trigger.

Cost & TCO

Every dead topic still rents disk, memory, and metadata.

Kafka topic sprawl is not a cosmetic problem. Storage, partitions, broker memory, controller state, schema subjects, and managed-Kafka per-resource billing all compound. A 4,000-topic cluster where 25% is dead is a real line item on the platform budget — and a real performance ceiling on the controller.

Storage

Disk is the obvious one

Stale and oversized topics keep retention windows full of records nobody reads. On managed Kafka, that's a direct GB-month bill; on self-managed, it's EBS / SSD provisioning, replication bandwidth across min.insync.replicas, and longer recovery times after broker restarts.

~10MB heap / partition
leader-side overhead
Partitions

Every partition has overhead

Each partition consumes leader-side broker memory (page-cache reservation, index files, follower-fetch state), controller metadata records, replication threads, and file-handle budget. Over-provisioned topics quietly cap the cluster's total partition headroom — often the first scaling wall hit before disk.

O(N)metadata records
replayed on every failover
Controller

KRaft / controller load

Every topic, partition, replica assignment, and config override is a record in the metadata log. Dead topics still get described, fetched, and replayed on controller failover. A leaner cluster means faster recovery, faster metadata propagation, and lower KRaft snapshot pressure.

Schema Registry

Orphan subjects rent storage too

Schema Registry providers bill per stored subject (Confluent Cloud) or per managed registry instance with storage caps (AWS Glue, Apicurio). Topics whose producers were decommissioned years ago still have -key and -value subjects taking up the budget.

Monitoring

Observability scales with topics

Every topic emits per-partition metrics: bytes_in, bytes_out, messages_in, replica lag, ISR shrink/expand, log-end-offset. At N topics × M partitions × K brokers, the cardinality bills your Prometheus, your Datadog, and your Splunk indexing tier — for series nobody plots.

Governance

The human bill

Every governance review, every ACL audit, every onboarding doc, every "do we still own this?" Slack thread is amortized across the topic count. Halving the surface halves the cognitive overhead for the platform team — that's where the real, hard-to-quantify savings live.

billed per topic-partition

How it works

A read-only scan. That's it.

Single static binary connects with admin-only ACLs, calls broker introspection APIs, computes the ATTIC Score, renders the verdict. No producer client is compiled into the binary. No record contents ever leave the broker.

What you get

Five sub-signals. One score. A verdict you can defend.

The ATTIC Score is a weighted heuristic, not a probability. Each sub-signal carries an evidence level. Weak evidence caps the verdict — you never delete on hearsay.

Hand-reviewable shell. Strict inclusion: only LIKELY_UNUSED with full evidence. Per-topic --describe preflight, no Kafka native dry-run, no automation — you keep the trigger.

Cluster support

Works on every Kafka you might be running.

Same scoring math everywhere. When a vendor restricts a broker API, the affected sub-signal is marked UNKNOWN and its weight is redistributed across the others. The score still computes — the JSON snapshot records the degraded evidence for downstream tooling.

Open methodology

The ATTIC Score is a public spec, not a black box.

Every sub-signal, every weight, every evidence transition, every verdict cap is documented in a versioned spec. Snapshots pin the spec version they were computed under, so historical diffs stay comparable.

    Days since the most recent record.
    w 0.30
  • T

    w 0.20
  • T

    w 0.10

Evidence levels: KNOWN, ESTIMATED, UNKNOWN. Verdict caps — any ESTIMATED → max Candidate; COMPACTED or REMOTE_STORAGE → max Inspect; APPEARS_NEVER_USED without PURGED → max Candidate.

Read the full spec → docs/attic-score-spec-v1.0.md. Methodology is published under CC BY 4.0 so other Kafka tooling can adopt it independently of this implementation — the way DORA, SPACE, and similar measurement frameworks are shared. 

# attic-score-spec-v1.0.md (excerpt)

verdict_bands:
  likely_unused: >= 90
  candidate:     70 .. 89
  inspect:       40 .. 69
  active:        < 40

activity_curve: # days → sub-score (0..100)
  0d:   0
  30d:  25
  90d:  55
  180d: 80
  365d: 100

verdict_caps:
  - when: "any sub-signal evidence == ESTIMATED"
    max: CANDIDATE
  - when: "flag includes COMPACTED"
    max: INSPECT
  - when: "flag includes REMOTE_STORAGE"
    max: INSPECT
  - when: "APPEARS_NEVER_USED && !PURGED"
    max: CANDIDATE

weight_redistribution: # when a signal is UNKNOWN
  policy: proportional
  note:   "score still computes from one fewer signal"
FAQ

Honest answers.

Is it safe to run on production?

Yes. kafka-attic is read-only by construction. The binary uses franz-go configured as an admin/consumer client only; the producer client is statically excluded at compile time and a CI test asserts that no producer-capable code path is reachable. Scans use bounded concurrency (configurable) and back off on throttling. For multi-thousand-topic clusters, a --sample mode lets you size impact first.

Will kafka-attic delete topics for me?

No. The cleanup script is a generated shell artifact you review and run by hand. kafka-attic never executes a destructive command itself. The verdict is LIKELY_UNUSED, never SAFE_DELETE. Topics with MISSING_SIGNAL, COMPACTED, REMOTE_STORAGE, or UNKNOWN evidence are excluded from the cleanup script. The script ships with per-topic preflight --describe commands and a warning banner — Kafka has no native dry-run for topic deletion.

Does it work on Confluent Cloud / MSK Serverless?

Yes, with degraded Tonnage. DescribeLogDirs is restricted on both, so kafka-attic skips the Tonnage sub-signal and redistributes its weight. Confluent Cloud topics on tiered storage or infinite retention also receive REMOTE_STORAGE, capped at INSPECT. The four other sub-signals collect normally.

What Kafka permissions does it need?

Read-only ACLs: Describe on Cluster, Topic, and Group; DescribeConfigs on Topic; DescribeLogDirs on Cluster (optional — enables Tonnage). For Schema Registry, a read-only API key with Subject:Read. For MSK IAM: kafka:DescribeCluster, kafka-cluster:DescribeTopic, kafka-cluster:DescribeGroup, kafka-cluster:DescribeTopicDynamicConfiguration, kafka-cluster:Connect. The scan reports observed permissions so you can audit access after the fact.

How is the score calibrated?

Defaults (Activity 0.30, Tenancy 0.20, Tonnage 0.10, Intent 0.15, Consumption 0.25) come from an anonymized calibration dataset gathered during the design phase, weighted toward signals reliably available on every conformant cluster. All weights, thresholds, and the activity curve are overridable in kattic.yaml. Every JSON snapshot pins the attic_spec_version it was computed under, so historical snapshots remain reproducible and comparable via kattic diff.

What about seasonal topics that fire monthly?

Seasonality detection is out of scope for v1.0. Two mitigations exist: (1) the verdict is conservative — Consumption goes to 0 the moment records exist, so topics with current data are capped well below LIKELY_UNUSED; and (2) the optional history database (history.enabled: true) accumulates past scans so the v1.1 seasonality detector can pick them up. Until then, add monthly/quarterly producers to exclude_patterns with effect: mark_protected.

Why is the methodology spec under a different license than the code?

The code is Apache 2.0; the methodology spec (docs/attic-score-spec-*.md) is CC BY 4.0. The spec is a piece of documentation meant to be cited, forked, and re-implemented by other Kafka tooling — CC BY 4.0 is the standard license for that kind of artifact. The dual licensing is intentional and reflects how DORA, SPACE, and similar measurement frameworks are published.

What's next: scan your first cluster

A minimal kattic.yaml needs a bootstrap and an auth block. Secrets come from environment variables — the file is safe to commit. 

cluster:
  name: prod
  bootstrap: broker-1.kafka.acme.com:9092
  auth:
    type: scram
    mechanism: SCRAM-SHA-512
    username_env: KAFKA_USER
    password_env: KAFKA_PASS
# run the scan
kattic scan --cluster ./kattic.yaml
From one-shot to continuous

When the question changes, the runtime changes.

kafka-attic is one-shot by design. It produces a snapshot you can run from a laptop, a CI job, or a cron container. It does not watch a cluster, it does not page an owner, it does not coordinate cleanups across teams, and it does not keep an audit trail of who deleted what. Those are continuous-monitoring concerns and they require a different runtime — one with a persistent process, RBAC integration, and durable state.

When a team's question moves from "what does our attic look like?" to "alert me when a topic becomes stale, ping the owner, and require an approval before deletion", that is the work Conduktor Console Insights does. Same evidence model, continuous evaluation, RBAC-aware owner routing, approval workflows, multi-cluster aggregation, cost attribution. kafka-attic stays OSS and stays one-shot. Console picks up where the one-shot ends.
